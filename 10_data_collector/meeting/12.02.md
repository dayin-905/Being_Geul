## **1. 개발 및 아키텍처 방향성 (HTML vs React)**

- **현재 상황:** HTML/CSS 기반의 프로토타입이 일부 구현되어 있음.
- **이슈:**
    - **React 도입 여부:** 인터랙션(애니메이션, 스와이프 등) 구현을 위해 React 도입이 필요하다는 의견이 있었으나, 현재 팀 역량 및 일정 상 React 학습 곡선과 초기 세팅(폴더 구조 변경 등)에 대한 부담이 있음.
    - **코드 구조:** HTML 기반 코드에서 React로의 전환 시 폴더링 구조 및 문법 차이로 인한 혼란 우려.
- **결정 사항:**
    - **HTML/JS 유지:** 2026년 1월 16일 마감 일정을 고려하여, React로 전면 전환하기보다는 현재 익숙한 **HTML/CSS/JS 기반으로 프로토타입을 완성**하는 방향으로 잠정 결론.
    - **Figma Dev Mode 활용:** Figma Dev Mode를 사용하여 HTML/CSS 코드를 추출하고, 이를 기반으로 개발 진행. (React 변환은 추후 고려)
    - **네이밍 규칙 통일:** 클래스명 등 네이밍 규칙(Naming Convention)을 사전에 정의하여 추후 React 전환 가능성을 열어두고 코드 충돌 방지. (부모 클래스 의존성 제거 등)

## **2. 데이터 수집 및 API 활용 (기업마당 vs 청년 지원)**

- **현재 상황:** '기업마당' API를 활용하여 데이터를 수집 중이나, '청년 지원 정책'이라는 프로젝트 취지와 맞지 않는 데이터가 다수 포함됨.
- **이슈:**
    - 기업마당 데이터는 '기업' 중심이라 대학생, 취준생 등 일반 청년 대상 정책이 부족함.
    - API 별로 데이터 분류 체계가 상이하여 DB 통합 시 문제 발생 가능성.
- **결정 사항:**
    - **타겟 확장 고려:** '청년'이라는 타겟을 39세 이하 등으로 넓게 정의하여 기업마당 데이터를 활용하되, **'청년' 키워드 외에 다양한 지원 정책(창업 등)을 포괄**하는 방향으로 데이터 수집 범위 조정 논의.
    - **데이터 필터링:** 수집된 데이터 중 유효한 데이터만 선별하기 위한 로직(우선순위, 매칭 로직 등) 수립 필요.
    - **추가 API 검토:** 기업마당 외에 청년 정책 관련 추가 데이터 소스(API) 발굴 및 DB 통합 방안 모색 (영우님 담당).

## **3. 데이터베이스(DB) 및 백엔드 설계**

- **이슈:**
    - API에서 불러온 데이터를 그대로 사용할지, 별도 DB에 저장하여 사용할지에 대한 논의.
    - API 호출 횟수 제한 및 속도 문제로 인해 자체 DB 구축이 필요함.
- **결정 사항:**
    - **자체 DB 구축:** API 데이터를 주기적으로 수집(크롤링/API 호출)하여 **자체 DB에 저장**하고, 서비스는 자체 DB를 참조하는 방식으로 확정. (안정성 및 속도 확보)
    - **데이터 최신화:** 마감일이 지난 정책은 자동 제외하거나, 주기적으로 데이터를 갱신(Update)하는 로직 구현 필요.
    - **중복 방지:** 데이터 수집 시 중복 데이터(Key 값 기준) 처리 방안 마련 필요.

## 1. 주요 논의 사항

### 🤖 개발 방법론 및 AI 활용 전략 (핵심)
* **80:20 법칙:** AI를 활용해 프로젝트의 **80% 수준까지 빠르게 구축**한 뒤, 다음 단계로 넘어가야 함.
* **코드 수정 역량 (Code Handling):** 남은 **20%의 디테일과 완성도(Customization)**를 높여 내 마음에 쏙 들게 만들기 위해서는, 개발자가 직접 **소스 코드를 수정하고 핸들링할 수 있는 능력**이 필수적임.
* **프롬프트 전략:**
    * 일반화된 완벽한 프롬프트를 제작하는 데에는 상당한 시간이 소요될 것으로 예상됨.
    * 프롬프트 입력 시 **사용하는 템플릿의 버전을 명확히 지정**해 주는 것이 결과물의 품질과 정합성에 매우 중요함.

### 💻 Frontend 기술 스택 (HTML/JS vs React)
* **결정:** React 도입보다는 **HTML/JS (Vanilla JS) 기반으로 진행**하기로 최종 결정.
* **기술적 근거:**
    * **"동적인 모든 기능은 JavaScript로 구현 가능하다."**
    * React는 결국 JavaScript의 라이브러리일 뿐이며, 현재 팀의 상황에서는 Vanilla JS로 로직을 짜는 것이 학습 및 구현 효율 면에서 더 나음.
    * React 도입 시 초기 세팅(폴더 구조 변경 등) 비용이 너무 큼.

### 📱 UI/UX 및 인터랙션
* **구조:** **원페이지 스크롤 + 상단 탭(Main, My Page, About Us)**을 이용한 페이지 전환 방식(SPA 형태) 확정.
* **네비게이션:** 우측 플로팅 네비게이션(스크롤 스파이) 기능은 구현 복잡도 및 효용성 문제로 **제외**.
* **웹 레이아웃:** 웹 환경에서는 모바일 앱처럼 보이도록 **중앙 정렬 컨테이너(Max-width 450px)** 내에서 작동하도록 구현.
* **애니메이션 최적화:** 성능 이슈 방지를 위해 로딩 시에만 Lottie를 재생하고, 스와이프 인터랙션 시에는 정적 이미지를 사용.

### 💾 데이터 처리 및 백엔드
* **소스:** 기업마당 API + 청년 정책 데이터.
* **필터링(ETL):** **N8N + Python** 파이프라인을 통해 기사/정책 분류 및 필터링 로직 구축 완료.
* **DB:** **PostgreSQL** 활용. (기존 논의된 Supabase에서 변경)
* **동기화:** 비회원 상태의 데이터(LocalStorage)를 로그인 시 DB로 이관(Upsert)하는 로직 구현.

### 🛠 기타 기능
* **Contact Us:** 보안(스팸 봇 방지)을 위해 **로그인한 사용자만 작성 가능**하도록 제한.
* **홍보 영상:** 트래픽 관리를 위해 YouTube/Vimeo 임베드 방식 권장.
